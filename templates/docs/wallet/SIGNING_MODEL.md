# Signing model (Floou mock / signing client)

Два **независимых криптоконтура**. Не объединять ключи в одну «универсальную» сущность без явной спецификации.

## 1. Контур Arweave (Data Item / ANS-104)

**Роль:** подпись материала для шага crystallization на стороне uploader (`crystalize` или эквивалент в вашем стеке).

**Классы переменных (имена — примеры, не секреты):**

| Класс | Назначение |
|-------|------------|
| Режим подписи | Выбор между локальной валидацией/mock, JWK (RSA), отладочным режимом — как в эталоне `ARWEAVE_SIGN_MODE` |
| JWK / путь к ключу | Только для режимов, где нужен реальный материал подписи; в примерах — пустые placeholder |
| URL uploader | Базовый URL сервиса, принимающего подписанный payload |

**Связь:** signing client получает payload от ноды, подписывает в этом контуре, отправляет на uploader.

## 2. Контур EVM

**Роль:** подпись транзакции/сырого payload для шага, который нода принимает через `submit` (или аналог).

**Классы переменных:**

| Класс | Назначение |
|-------|------------|
| RPC URL | Должен соответствовать сети, с которой работает нода (`chain_id` согласованность) |
| Приватный ключ подписанта | Dev-only в `.env`; в проде — вне репозитория / secret manager |

**Связь:** не обязательно прямое подключение к Hardhat; типично RPC + нода как broadcast/mediator.

## 3. Identity

- Один **логический пользователь** для POC: идентификатор пользователя на ноде (заголовок/поле вроде `X-User-Id`) согласован с activity flow.
- Multi-tenant модель в этом шаблоне **не** предполагается — расширение отдельным решением.

## 4. Wallet-auth (опционально)

Нода может выдавать challenge и принимать подпись для выдачи сессии (Bearer). Классы переменных для клиента: base URL ноды, учётные данные mock — см. `wallet-mock.env.example`.
